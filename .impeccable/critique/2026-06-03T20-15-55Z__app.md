## Anti-Patterns Verdict

The app does not look like generic AI marketing output. It uses a restrained product UI, consistent dark surfaces, simple grids, and familiar controls. The main issue is not visual excess; it is operational incompleteness. The interface looks plausible before it behaves like a reliable household utility ledger.

Browser visualization: attempted with the current Browser plugin. Browser runtime was found, but navigation to local pages failed with CDP navigation errors/timeouts. No reliable visual overlay is available.

## Overall Impression

The product direction is sound: a private utility dashboard split into light and water. The biggest opportunity is to shift from "view-only dashboards plus create modals" to a complete month-closing workflow where every talão/conta can be entered, verified, corrected, marked paid, and understood at a glance.

## What's Working

1. The service split is correct. `/servicos` separates luz and água cleanly, reducing conceptual collision.
2. The water dashboard's month grouping matches the user's mental model better than a flat history list.
3. The shared visual language is coherent: dark surfaces, compact cards, clear typography, and stable navigation.

## Priority Issues

### [P1] Water dashboard is missing operational actions

Why it matters: The monthly view shows each ID, but the user cannot mark a talão as paid, edit a wrong value, delete a duplicate, or attach/change the file from the row. That forces correction through hidden/database paths.

Fix: Add inline actions per talão: `Marcar pago`, `Editar`, `Substituir arquivo`, `Excluir`. Keep create modal, but make monthly rows operational.

Suggested command: `impeccable harden`

### [P1] Empty and setup failure states are not actionable

Why it matters: `/agua` depends on `contas_agua` and `contas-agua`. If SQL is not applied, users will likely see generic Supabase failure behavior or an empty screen that appears valid.

Fix: Detect missing table/storage errors and show explicit admin-facing setup guidance. In empty months, show "Cadastrar Junho/2026 para Casa X" actions.

Suggested command: `impeccable onboard`

### [P1] Mobile navigation drops important routes

Why it matters: Bottom nav has only five items. `Residências`, `Analytics`, and `Configurações` are partially hidden or only reachable through top bar/settings. For a phone-first household workflow, this causes recall burden.

Fix: Add a "Mais" destination or service-aware navigation grouping. Do not keep adding items to the bottom nav; group secondary routes.

Suggested command: `impeccable adapt`

### [P2] Information architecture mixes domain and system terms

Why it matters: "Competências", "Analytics", "Faturas", "Luz", "Água", and "Serviços" coexist at the same nav level. Some are workflows, some are domains, some are reports.

Fix: Group under domains: `Luz` contains Dashboard, Competências, Faturas, Analytics; `Água` contains Talões, Matrículas, Histórico. Keep `/servicos` as mode switch.

Suggested command: `impeccable clarify`

### [P2] Overuse of card grids reduces scan efficiency

Why it matters: Cards are coherent but repetitive. Monthly water sections nested with 3 mini-cards per month can become long quickly.

Fix: Use a compact monthly table/grid for água: columns for Casa, Matrícula, Valor, Responsável, Status, Ações. Preserve cards for summary only.

Suggested command: `impeccable layout`

## Persona Red Flags

Alex, power user: Cannot quickly close a month of water bills. No batch creation for the 3 talões, no keyboard shortcut, no direct mark-paid action in the dashboard.

Sam, accessibility-dependent user: Focus states exist, dialogs have ARIA, but status is heavily color-coded and some state changes after save depend on refresh without live announcement.

Casey, mobile user: The FAB exists for light readings, but water's primary per-month actions are top-positioned in a modal. Bottom navigation also omits some required destinations.

Riley, stress tester: Duplicate/missing table/storage states are not handled as first-class UI states. Long matrícula strings, many months, and wrong uploaded file cases are not visibly constrained.

## Minor Observations

- `MetricCard` defines `trend` but does not render it.
- `StatusBadge` uses color labels well, but lacks icons or text patterns that make state recognizable beyond color.
- The service hub repeats the same card structure twice. Acceptable now, but it will not scale if more services are added.
- Water status can be set to `pago` without a payment date; the UI then says "Pago Sem data".
- "Analytics" should likely become "Relatórios" or "Histórico" for a household utility product.

## Questions to Consider

- Should água be optimized for monthly closing, not just monthly viewing?
- What is the fastest path for entering all 3 talões for Junho/2026?
- Should "Competências" remain visible to the user, or is it an internal accounting term?
- What happens when a talão was entered wrong after payment?
